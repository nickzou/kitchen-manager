Spec: Recursive recipe nutrition, cost, and shopping list

Kitchen Manager · Draft

Drafted2026-05-28
Shipped—

Problem

When a recipe's ingredient is a producible product (a product with no nutrition of its own, where the calories live on the source recipe that produces it), the client-side nutrition rollup misses those ingredients.

Concrete example from production, meal plan 2026-05-26:

ItemClient-side cal
Buttered Bagel and Coffee~292
Pad Kra Pao (Meal-Prepped)~462
Protein Shake~376
Spaghetti con broccoli aglio e olio~1066
Sum (client)2,196
Day total (server)2,981

The 785-calorie gap shows up because the meal-prepped dish references a prepped product whose nutrition is derived from a source recipe — the server's nutrition-summary route follows that link, but getRecipeNutrition in src/lib/recipe-utils does not.

The cost rollup has the same shape of bug (getRecipeCost uses getAvgUnitCost on stock; producible products with no stock get skipped). Same fix, same place.

The shopping list has a related-but-different gap: when a planned recipe needs a producible product (e.g. "Pad Kra Pao mix") and there's no stock, the current ingredient-summary surfaces it under a "producible" bucket with a "Make from recipe" hint — but it does not auto-expand into the source recipe's ingredients (the things you actually have to buy). The user has to remember to open the source recipe and add its ingredients manually.

Goal

  • Per-recipe and per-meal-plan-entry nutrition shown in the UI matches the server's day total to within rounding.
  • The fix is in shared utilities so it benefits the recipe detail page, the meal-plan entry modal, and the per-day rollup component the same way.
  • Cost gets the same treatment in the same PR — same data plumbing, no point doing it twice.
  • Shopping list: when a planned producible product has no stock, auto-expand the shortfall into the source recipe's ingredients. Nets out against existing stock of those ingredients the same way regular ingredients do.

Approach

1. Shape: one helper, one extra input

Extend getRecipeNutrition and getRecipeCost to accept an optional derivedByProduct map keyed by product id. When an ingredient's product has no own nutrition/cost, fall back to the derived value for that product before giving up. 

type Derived = { calories, protein, fat, carbs, baseAmount, baseUnitId };
const derived = derivedByProduct.get(ing.productId);
if (!product.calories && !product.protein && ... && derived) {
  // treat `derived` as if it lived on the product, normalised the same way
}

This mirrors the existing deriveProductNutrition helper the server already uses; we'd extract it into src/lib/recipe-utils/ and use it on both sides so the math is identical.

2. New hook: useDerivedNutritionByProduct

A React Query hook that returns the same per-product derived nutrition map the server builds. Backed by a new GET /api/products/derived-nutrition endpoint that runs the recursion once and returns: 

{
  [productId]: {
    calories: number, protein: number, fat: number, carbs: number,
    baseAmount: number, baseUnitId: string | null,
    complete: boolean
  }
}

The server already computes this per request inside nutrition-summary; extracting it lets the dashboard, recipe detail page, meal-plan entry modal, and shopping list all share one cached source.

Why a new endpoint instead of expanding the existing nutrition-summary? nutrition-summary is keyed by date range and returns per-day rollups. We want per-product derived values that we can reuse for the recipe detail page (no date) and the per-entry modal. Keeping them separate also means the day-rollup endpoint can stay narrow and fast.

3. Recursion depth

The server today does one level of derivation — if a source recipe itself contains another producible product, that secondary product is not expanded. We keep that limit for now (rare in practice; an unbounded recursion would also need a cycle guard).

If the user does build a chain (recipe A produces P1, recipe B uses P1 and produces P2, recipe C uses P2), we will under-count C, the same way we under-count today. Out of scope for this PR; would be a follow-up if/when this surfaces.

4. Cost parallel

Mirror the same pattern: useDerivedCostByProduct hook + a derivedByProduct input to getRecipeCost. The derived cost of a produced product comes from averaging the cost of the source recipe's ingredients per unit produced.

Same one-level limit, same first-wins rule (if multiple recipes produce a product, pick the first — matches the server's existing behaviour).

5. Shopping-list expansion

Refactor ingredient-summary so that when a planned producible product is short on stock, the shortfall is expanded into the source recipe's ingredient list rather than leaving the user to follow a "Make from recipe" hint by hand.

  • For each producible product whose need > stock, compute the deficit (the amount you'd have to "make"), then divide by the source recipe's producedQuantity to get the number of source-recipe servings needed.
  • Multiply each source-recipe ingredient by that factor and roll it into the same ingredients aggregation the rest of the summary already builds (so it nets against any in-stock ingredients automatically).
  • If the source recipe itself uses another producible product without stock, stop there — same one-level limit as nutrition/cost. The "Make from recipe" affordance still appears for any unresolved producible chains so the user has visibility.
  • The "producible" bucket in the response stays as-is for products the user genuinely wants to make themselves (i.e., still has source-recipe stock). The expansion is opt-in based on whether source-recipe ingredients are buyable.
Side effect: this closes ClickUp 86e19mwm6 ("Auto-expand source-recipe ingredients into shopping list when producible stock is short") and partially addresses 86e19mw30 ("Recursive producible products in shopping list / nutrition"). The latter stays open for the unbounded recursion case if we ever pursue it.

Tests

  • Unit: getRecipeNutrition with a producible product and a derived map returns the expected calories (uses the derived value).
  • Unit: same recipe with derived map omitted falls back to today's behaviour (undercounts) — to prove the new path is the only one that changes behaviour.
  • Unit: deriveProductNutrition handles the "no ingredients" and "source recipe with no produced quantity" cases gracefully.
  • API: GET /api/products/derived-nutrition returns the expected derived value for a known fixture (recipe → product → ingredients with nutrition).
  • Integration: per-entry modal calorie display matches the dashboard's day total for a day with at least one meal-prepped entry.
  • API: GET /api/meal-plan-entries/ingredient-summary with a planned producible product and zero stock returns the source recipe's ingredients in the ingredients array, scaled by the deficit.
  • API: same call with enough source-recipe stock that the deficit can be made still surfaces a "producible" entry (no double-counting).

Decisions

  1. Endpoint shape: two separate endpoints — one for derived nutrition, one for derived cost.
  2. Server refactor: yes — nutrition-summary moves to the new shared helper in this PR.
  3. Multi-source-recipe products: keep first-wins, matching today's server behaviour.
  4. "Complete" flag: surface in the UI with a "*" partial marker so users know when a number is partial.

Out of scope

  • Unbounded recursion / cycle detection. One level only.
  • Recomputing historical day totals — this is a display fix, not a data fix.
  • Letting the user pick a canonical source recipe per product (e.g., "use this recipe to compute Pad Kra Pao mix's calories"). The first-wins behaviour stays; a chooser is a future feature.
  • Cost: per-stock-entry pricing precision (averaging across entries already handled).
