Spec: Meal Plan recipe modal & entry detail

Kitchen Manager · Draft

Drafted2026-05-22
Shipped2026-05-22

Problem

Two related rough edges on the Meal Plan page:

  1. The recipe picker is an inline Combobox jammed inside each grid cell. On desktop the cell is ~120px wide; on mobile it's bigger but still cramped. You only see a name — no image, no categories, no signal of whether you have the stock to actually cook it.
  2. Once a recipe is planned, the entry popover only exposes servings ±, Cook, and Delete. The recipe name is plain text — no way to jump to the recipe page from the plan. And the popover hides the two things you'd most want to see when scheduling: what will this cost and how many calories / macros.

Goal

  • Picking a recipe to plan should feel like browsing a small catalogue, not typing into a 120px input.
  • Looking at a planned entry should answer "what is this, how much does it cost, what are the macros, and can I cook it now?" without leaving the page — with a clear way to jump to the full recipe.

Approach

Two new pieces, both reusing existing data hooks:

1. Add-recipe modal (MealPlanRecipePickerModal)

Replace the inline Combobox in MealPlanCell with a + button. Clicking it opens a Modal scoped to the cell's (date, mealSlotId) showing:

  • Search field that filters the recipe list as you type.
  • Scrollable list of recipe rows. Each row shows:
    • Thumbnail (recipe.image) or placeholder square.
    • Recipe name + category tags.
    • Servings / prep time / cook time inline.
    • Stock availability badge from useRecipeAvailability — CircleCheck if sufficient, CircleX if deficit, nothing if no ingredients.
  • One click on a row commits the add and closes the modal.

2. Recipe entry detail (replace MealPlanEntryPopover)

The current floating popover is too small for what we want to show. Replace it with a Modal that renders the same info plus:

  • Recipe thumbnail + name (the name is a Link to /recipes/$id).
  • Quick facts block:
    • Calories + P/F/C macros from getRecipeNutrition, scaled to the entry's servings.
    • Estimated cost from getRecipeCost, scaled the same way. Show a * if the cost is partial.
  • Existing controls: servings ±, Cook / Undo Cook, Delete.
  • "View recipe →" footer link as a second affordance in case the user misses that the title is also a link.
Why a Modal, not an inline expander or a bigger popover? The popover's positioning logic already has to flip above/below depending on available space. Adding nutrition + cost + a thumbnail makes it tall enough that it'll regularly clip on mobile. A modal centers content, scrolls cleanly, and lets us reuse the existing Modal primitive.

Data sources (no new hooks needed)

  • useRecipes(), useRecipeCategories() — list + tags for the picker.
  • useRecipeAvailability() — per-recipe stock signal.
  • useRecipeIngredients(recipeId), useProducts(), useStockEntries(), useUnitConversions(), useAllProductConversions() — inputs to getRecipeNutrition and getRecipeCost for the entry detail modal.

Fetching these only when the detail modal is open keeps the meal-plan landing query graph the same as today.

Tests

  • MealPlanRecipePickerModal: renders when triggered, filters by search, picks a recipe and fires onPick(recipeId), closes on backdrop click.
  • MealPlanEntryDetailModal: renders nutrition + cost rows, the title links to the recipe page, Cook / Undo Cook / Delete fire the correct callbacks, servings ± updates locally and calls onUpdateServings.
  • Light integration check on the meal-plan page: clicking + opens the picker, clicking a recipe closes it and the entry appears in the cell.

Decisions you need to make

  1. Mobile presentation: same Modal on mobile (full-bleed sheet feel), or a bottom-sheet variant for the picker? Default is "use the existing Modal, revisit if it feels off."
  2. Empty state in the picker: if the user has no recipes yet, show a "No recipes — create one" link to /recipes inside the modal?
  3. Cost rounding in the detail modal: same 2-decimal currency formatting the calendar already uses? (Assumed yes.)

Out of scope

  • Adding a recipe to multiple slots/days in one open. One modal open = one entry.
  • Creating a recipe from inside the picker modal. The "No recipes" link to /recipes is the escape hatch.
  • Showing per-ingredient detail in the entry modal — that's what the recipe page itself is for.
  • Reorder / drag-and-drop between slots.
