Spec: Prevent duplicate brands (and similar)

Kitchen Manager · Draft

Detected2026-05-22
Resolved—

Problem

The brand "Lactantia" exists twice in production, created 7 seconds apart — most likely from a double-click on the create form. One copy has stock entries attached; the other is orphaned. There is no DB-level or API-level guard against this. The same issue can recur for any user-named lookup table (brands, stores, quantity units, categories).

Goal

Make it impossible for a user to end up with two brands of the same name in their account. Same fix should apply to other user-named lookup tables that share this shape.

Approach

Two-layer defence, ship in that order:

  1. DB-level unique constraint on (user_id, lower(name)). Authoritative, race-proof, language-agnostic. POST returns 409 with a clear error message.
  2. Client-side guard: disable the create button while the request is in flight, and surface a soft "already exists" hint inline as the user types (using the brands list the page already has cached). UX polish on top of the DB layer.
Case sensitivity: product already has a unique constraint on (user_id, name) but it's case-sensitive, so "Salt" and "salt" can both exist. Proposal: switch to case-insensitive for this new work (unique index on lower(name)). Decide whether to apply the same change to the existing product constraint.

Migration strategy

Two concerns: (a) creating the new unique indexes, (b) cleaning up existing duplicates so the index can be created.

1. Audit existing data first

Before generating the migration, run a SQL query (or one-off script) to find any duplicates that would block the new index:

SELECT user_id, lower(name), count(*)
FROM brand
GROUP BY user_id, lower(name)
HAVING count(*) > 1;

Repeat for each affected table. If anything is found, surface it to the user for manual merge before running the migration.

2. Migration shape

One Drizzle migration per table (or combined), using a functional unique index:

CREATE UNIQUE INDEX brand_userId_name_unique
  ON brand (user_id, lower(name));

Generated via drizzle-kit generate --name=add-unique-brand-name.

API behavior

POST and PUT handlers on each affected resource catch Postgres error code 23505 (unique violation) and return a 409 with a clear error message. The product handler already does this — pattern to copy: 

try {
  [created] = await db.insert(brand).values({...}).returning();
} catch (err) {
  if (err.code === "23505") {
    return json(
      { error: "A brand with this name already exists" },
      { status: 409 }
    );
  }
  throw err;
}

Client UX

  • Disable submit while pending — already partly done elsewhere; ensure consistent.
  • Inline duplicate hint: as the user types in the create field, check the cached brand list (already in React Query). If a case-insensitive match exists, render a small "Brand already exists" message next to the input and gray out the submit. Cheap to wire because the list is already loaded.
  • 409 toast on submit: if the inline check missed (race, list cache stale), surface the server's 409 message via the existing toast system.

Tests

  • Endpoint test: POST a duplicate (same name, different case) returns 409 with the expected error body.
  • Endpoint test: PUT renaming brand B to match brand A's name returns 409.
  • Endpoint test: POST a non-duplicate succeeds.
  • Schema test or repeat the above for each affected resource that gets the constraint.

Decisions you need to make

  1. Case sensitivity: case-insensitive uniqueness for the new constraints? (Recommended.)
  2. Existing product constraint: leave as case-sensitive, or migrate it to case-insensitive too (drops the old constraint, adds the new functional one)?
  3. Scope: ship for brands only first (small, focused PR), or cover brands + stores + units + categories + recipes in one PR? Same shape of change applied uniformly.
  4. Pre-migration duplicates: if the audit query finds existing duplicates beyond Lactantia, do we want a one-off cleanup script or surface them in-app for manual merge?

Out of scope

  • "Merge brand" UI flow for resolving duplicates that already exist — useful future feature, not needed for this fix.
  • Fuzzy-match duplicate detection ("Lactantia" vs "Lactancia") — case-insensitive only.
  • Trimming whitespace before comparison — could add, but a separate concern.
