Spec: Shopping list defaults to today → end of week

Kitchen Manager · Draft

Drafted2026-05-22
Shipped2026-05-22

Problem

Opening the Shopping List always defaults the range to start-of-week → end-of-week, honoring the user's weekStartDay setting. If today is Wednesday and the week starts Monday, the picker fills Mon–Sun, including two days that have already passed. Those past days drag in meals you've already cooked or skipped, padding the list with stuff you don't actually need to buy.

The picker also resets to the default on every navigation back to the page — even a trip to a recipe page and back wipes the manually chosen range.

Goal

  • First visit in a session: range is today → end of the current week, where "end of week" honors weekStartDay (i.e. getWeekStart(today, weekStartDay) + 6 days).
  • Subsequent visits in the same session: keep whatever range the user picked.
  • Behaviour is preserved across SPA navigations and page reloads within the same browser session.

Approach

1. Compute defaults from today, not from week start

Replace the current init block in shopping-list.tsx: 

// before
const [startDate, setStartDate] = useState(() =>
  toDateString(getWeekStart(new Date(), 1)),
);
const [endDate, setEndDate] = useState(() => {
  const end = getWeekStart(new Date(), 1);
  end.setDate(end.getDate() + 6);
  return toDateString(end);
});

with a helper that computes the new default:

function todayToWeekEnd(weekStartDay: number) {
  const today = new Date();
  today.setHours(0, 0, 0, 0);
  const end = getWeekStart(today, weekStartDay);
  end.setDate(end.getDate() + 6);
  return {
    start: toDateString(today),
    end: toDateString(end),
  };
}

Edge case: if today is the last day of the week, start ≡ end — that's correct, just one day's worth of plan.

2. Persist the user's chosen range for the session

Use sessionStorage (not localStorage) so the range is sticky across reloads and route changes within the session, but resets cleanly to the today-based default next session. Key: shopping-list-range, value: {"start": "YYYY-MM-DD", "end": "YYYY-MM-DD"}.

Initial state resolution order:

  1. If sessionStorage[shopping-list-range] exists and parses, use it.
  2. Otherwise compute todayToWeekEnd(weekStartDay).

When the user changes either date via DateRangePicker, write the new pair back to sessionStorage.

3. Reconcile with the existing settings-load effect

Today there's a useEffect guarded by a settingsAppliedRef that re-applies the week-start default once user settings finish loading. That effect becomes a no-op the moment we move to today-based defaults — the initial state doesn't depend on weekStartDay being known. Remove the ref and effect.

We do still need to wait for the first render before reading from sessionStorage if we want to stay SSR-safe; do the read inside a useEffect (same shape the existing useCheckedItems hook uses).

Tests

  • Unit test for todayToWeekEnd: given a known today + weekStartDay, returns the expected start/end pair (covering "today is week start", "today is mid-week", "today is week end").
  • Integration check: render the page with an empty sessionStorage and assert the picker fills with today → week end.
  • Integration check: pre-seed sessionStorage with a stored range, render, assert the picker fills with the stored values.
  • Integration check: changing the range writes back to sessionStorage.

Decisions

  1. Persistence scope: sessionStorage. Resets on browser close; we don't carry a stored range into a fresh session.
  2. Day flip in the same session: reset to today → end of week when the calendar moves past the day the range was saved. To detect this we also store a savedAt date alongside the range; if savedAt ≠ today on mount, we ignore the stored range and recompute defaults.
  3. Reset affordance: include a small "Reset" button next to the date picker that snaps the range back to today → end of week and updates sessionStorage with today's savedAt.

Out of scope

  • Changing the Meal Plan page's week-start behaviour — that one still anchors on weekStartDay.
  • Server-side persistence of the range — if we ever want it to sync across devices, that's a separate change.
  • Touching the existing useCheckedItems persistence (which is keyed by range and writes to localStorage) — left as-is.
