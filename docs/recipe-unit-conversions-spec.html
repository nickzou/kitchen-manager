Problem

Recipe ingredients display in whatever unit the recipe author chose — cups, tablespoons, ounces, pounds, mL, grams, kg. The conversion graph already exists in the app (per-product overrides plus global conversions), but it's only consulted invisibly during the cook flow. From the user's point of view, the recipe page is a static list.

Two concrete pains that motivate this:

  • A recipe calls for "1 cup of butter" but the user buys butter in grams (because that's how grocery stores sell it). They have to mentally convert or look it up to know how much to portion out.
  • A recipe calls for "200 g of flour" but the user thinks in cups. Same problem in the opposite direction.

The lead ICP (value-conscious home cook) is especially likely to have one preferred working unit per ingredient class — usually whatever their groceries are labelled in — and is least likely to want to do mental arithmetic.

Goal

  • Let the user see ingredient quantities in a unit other than the one the recipe author wrote, without leaving the recipe page.
  • Reuse the existing conversion graph (product-specific + global) — no new data.
  • Fail gracefully when no conversion is possible.
  • Don't add UI clutter to ingredients that don't benefit (e.g. "2 eggs" doesn't convert to anything useful).

UX options considered

Option 1 — Always-show the stock unit alongside the recipe unit (recommended)

Render every convertible ingredient as recipe-unit (stock-unit), inline. The stock unit is the product's defaultQuantityUnitId — i.e. how the user actually buys/stores it. If recipe unit and stock unit are already the same, nothing changes.

Recipe ingredient list, Option 1
✓ Butter 1 cup ≈ 227 g
✎ 🗑
✓ All-purpose flour 2 cups ≈ 240 g
✎ 🗑
✓ Salt 1 tsp ≈ 6 g
✎ 🗑
✓ Eggs 3 large
✎ 🗑

Pros: zero interaction required, always visible, lowest cognitive load. Matches the way recipes in magazines / cookbooks often print both metric and imperial.
Cons: slightly more visual weight on every line. Adds a small dependency on the conversion graph being complete (gaps fall back silently to "no secondary unit shown").

Option 2 — Recipe-level pill toggle ("Show in")

A single toggle near the recipe header lets the user pick what to display for everything at once. Three modes: recipe units (today's behaviour), my stock units (convert all ingredients to their product's default), and both (Option 1's display). The choice persists per-user in localStorage.

Recipe-level toolbar, Option 2
Show units in: Recipe My stock Both
✓ Butter 227 g
✎ 🗑
✓ All-purpose flour 240 g
✎ 🗑
✓ Salt 6 g
✎ 🗑

Pros: minimal per-row visual change; very clean when set to "my stock". User explicitly chooses the convention they prefer.
Cons: hides information unless the user finds and changes the toggle. Most users will leave it at the default and never benefit. Discoverability is the failure mode.

Option 3 — Click-to-cycle on the quantity

The quantity itself is clickable. Each click cycles to the next available unit for that ingredient. State is per-ingredient, not persisted.

Click-to-cycle, Option 3
✓ Butter 1 cup ⇄
✎ 🗑
✓ All-purpose flour 240 g ⇄ (was 2 cups)
✎ 🗑

Click the quantity to cycle: cup → tbsp → mL → g → cup …

Pros: discoverable through interaction, doesn't add always-on visual weight, fun.
Cons: click target on quantity competes with text-selection intent. Hidden until interaction — users won't know they can do it. Per-ingredient state is fiddly. Mobile (tap) is fine, but desktop users trying to copy a number will accidentally cycle.

Option 4 — Hover/long-press popup

Hover (desktop) or long-press (mobile) the quantity to see a small popup with all available conversions. Static display, no state to persist.

Hover popup, Option 4
✓ All-purpose flour 2 cups ▾
✎ 🗑
2 cups of flour
= 240 g
= 8.5 oz
= 32 tbsp

Pros: shows all conversions, not just one. No visual noise at rest.
Cons: hover behaviour doesn't work on mobile (long-press works but is unfamiliar). Information is hidden until interaction. Most users will never trigger it.

Recommendation

Ship Option 1. Always-visible, low-cognitive-load, matches the way recipes are typically printed in cookbooks and online. Failure mode is graceful: if there's no conversion path, the secondary unit just doesn't render — no error, no clutter, no warning. The user gets value on day one without any setup or discovery.

Reasoning against the alternatives:

  • Option 2 requires the user to find a toggle, which most won't. Value-conscious cooks aren't looking for power-user features.
  • Option 3 introduces click ambiguity (cycle vs. text-select) and hides the feature behind interaction.
  • Option 4 doesn't work cleanly on mobile, where most cooking-time use happens.

Future possibility: Option 2's pill toggle could be added later as a power-user preference on top of Option 1, with the default still being "both" (Option 1's behaviour). But ship Option 1 first.

Conversion target choice

Option 1 shows one secondary unit per ingredient. The question is: which? Three reasonable defaults:

  1. Product's default quantity unit (defaultQuantityUnitId). This is how the user stores the product — if they buy flour in grams, the conversion shows grams. Most useful for the cook-from-pantry workflow.
  2. Always metric (g / mL / kg / L). Predictable, but ignores user context.
  3. User-chosen preference (per-user setting). More flexible, more setup.

Recommendation: option (1) — convert to the product's default unit. It's the unit the user has already declared they think in for that product, and it makes the conversion immediately useful for "do I have enough?" mental math.

Edge cases

  • Same units already. Recipe says 200 g, product default is g → don't render the secondary.
  • No conversion path. Recipe says "3 large eggs" with unit large, no conversion to g → secondary doesn't render. No error toast, no warning — silent fallback. (We already use the same "no conversion" signal for the stock-availability StatusIcon.)
  • Ingredient has no productId. Skip. There's no default unit to convert to.
  • Scale factor. Conversion happens after the recipe-level scaling is applied. scaledQuantity → convert → render.
  • Rounding. Keep it simple: integer for grams, integer or 1 decimal for mL, 2 decimals for cups/tbsp/tsp. The ≈ prefix signals approximation so users don't expect 4-decimal precision.

Tests

  • Unit test for formatConversion rounding rules (g / mL / oz / cup / tsp).
  • IngredientRow renders ≈ X g when recipe unit differs from product default and conversion exists.
  • IngredientRow does NOT render the secondary when units match.
  • IngredientRow does NOT render the secondary when no conversion path exists.
  • Scaled quantity: 1.5x recipe with 1 cup butter renders "1.5 cups ≈ 340 g".

Decisions

  1. UX: Option 1 (always-show secondary inline). Acknowledged it may not be flexible enough long-term — a future Option 2 pill toggle can layer on top without breaking this.
  2. Conversion target: product's default quantity unit.
  3. Rendering style: ≈ 227 g in muted lagoon-deep colour (as shown in the Option 1 mockup).
  4. Propagation: recipe page + Cook picker. Not the shopping list (already in store units). The meal-plan entry detail modal doesn't render individual ingredients, so it's a no-op there.

Out of scope

  • Unit preferences per-user (Option 2's pill toggle). Possible future, not this PR.
  • Showing all available conversions in a popup (Option 4). Same — future polish.
  • Auto-suggesting better conversions when a user enters a new ingredient (e.g. "you typed 240 g; did you mean cups for this recipe?"). Different feature.
  • Conversion accuracy callouts (e.g. flour densities vary — "this conversion assumes AP flour at 120 g/cup"). For another time.
