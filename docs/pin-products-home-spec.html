Spec: Pin products to the home page

Kitchen Manager · Draft

Drafted2026-05-25
Shipped—

Problem

Day-to-day, the same handful of products get consumed over and over — coffee, soy sauce, soda. Every consume currently requires going to the Stock page, scrolling or searching to find the product, opening the accordion, and clicking the quick-consume button. For frequently used staples that's a lot of friction.

Goal

Let the user pin individual products so they show up as a quick-access list on the home page, with a one-click consume that mirrors the Stock view's quick-consume behaviour (deduct the product's default consume amount from the best FIFO entry).

Approach

1. Data

Add two columns to the existing product table: 

pinned: boolean default false not null
pinnedSortOrder: integer (nullable — manual order)

Per-user is implicit because product is already scoped to userId. A separate join table is overkill for a single-user flag; the column-on-row approach keeps reads cheap and avoids a second query for the dashboard.

Migration via drizzle-kit generate --name=add-product-pinned. New rows default to pinned=false; nothing to backfill.

2. API

Extend PUT /api/products/$id to accept pinned and pinnedSortOrder (both optional). No new endpoint needed; the existing pattern of "PATCH-style PUT" already handles partial updates field-by-field.

GET /api/products continues to return everything, so the dashboard can filter pinned === true client-side without a new request shape.

3. Pin / unpin affordance

A star icon button on each product's detail page (top right of the header) and a smaller version on each Stock accordion trigger row. Filled gold star = pinned; outline grey star = not pinned. Click toggles via the existing update mutation; toast confirms.

4. Home page: Pinned section

New section <Pinned> rendered above Expiring Soon on the dashboard. Each row mirrors the Stock view's trigger + action layout: product name, total stock, default-consume label (in parens), Consume button. Out-of-stock pinned products still appear, with the Consume button disabled and stock shown as 0.

Home page, Pinned section
Coffee 701 g (1 g)
Soda 408 mL (1 mL)
Soy Sauce ⚠ 0 mL (1 mL)

Consume action: same flow as handleQuickConsume in the Stock view — pick best FIFO entry, apply default-consume amount, do unit conversion if needed, toast with Undo. The function can be extracted to a shared hook (useQuickConsume) so dashboard and stock view stay in sync.

5. Ordering

Display order: pinnedSortOrder ASC NULLS LAST, name ASC. Until we build drag-to-reorder, new pins land at the end and alphabetical kicks in for unordered ties. Drag-to-reorder is a separate, larger feature; not in this PR.

Tests

  • API: PUT /api/products/$id with { pinned: true } sets the column; toggling back to false works.
  • Component: PinnedProducts renders only pinned products; sorts by pinnedSortOrder then name.
  • Component: clicking Consume on a pinned product calls the same mutation path the stock view uses.
  • Component: out-of-stock pinned products still render but Consume is disabled.

Decisions you need to make

  1. Pin affordance location: product detail page only, or also on each Stock accordion trigger? Trigger-row pinning is fastest to discover, but adds one more icon to a busy row. Default: both.
  2. Hide out-of-stock pinned products? Show them disabled (default), hide them, or move them to a collapsed "Out of stock" subsection?
  3. Empty state on home: when no products are pinned, hide the section entirely or show a "Pin frequent items from the Stock page" hint? Default: hide.
  4. Star vs pin icon: Star (yellow) or Pin (more literal)? Default: star — fits the "favourite" mental model better.

Out of scope

  • Drag-to-reorder pinned products — keep alphabetical order for now.
  • Pinning recipes or meal plans — same pattern could apply later, but not here.
  • Per-device pins — pins are tied to the user account, not the device/browser.
  • Search/filter on the pinned section — assumed small enough to fit on one screen.
