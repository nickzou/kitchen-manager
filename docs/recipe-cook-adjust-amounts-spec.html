Problem

Recipes are written with target amounts ("100 g rice") but the actual cook rarely matches them exactly. You scoop 120 g of rice. You eyeball the salt. You're out of cilantro so you use parsley. None of those decisions are wrong — they're the reality of cooking. But the app's Cook flow consumes the recipe's amounts, not the actual amounts, so the stock ledger drifts from reality with every meal.

For the value-conscious ICP especially, this drift compounds. If the recipe says 100 g but you always use 120 g, your projected grocery costs are 20% low and your "do I have enough" availability check tells you there's stock when there isn't. The app is technically correct and practically wrong.

Goal

  • Let the user adjust per-ingredient amounts at cook time without changing the saved recipe.
  • Adjustments deduct the actual amount from stock, not the recipe's amount.
  • Recipe stays unchanged for future cooks — this is a per-cook tweak, not a permanent edit.
  • Discoverable in the natural Cook flow (not buried under a settings page).
  • Skippable — the existing one-click Cook path should still work for users who don't need to tweak.

UX options

Option 1 — Pre-cook adjustment modal (recommended)

Cook button opens a modal listing every ingredient with editable quantity fields pre-filled with the recipe-scaled amount. The user can tweak any of them, see the converted stock-unit display update in real time, and see availability flags. Click "Cook" in the modal to consume the adjusted amounts.

Pros: single review-and-confirm step matches kitchen workflow (you're at the counter, about to cook, about to measure). All adjustments visible at once. Reuses CookPicker's group-selection pattern for recipes with ingredient groups (just embedded in the same modal). The "↺ 100g" reset button per row makes undoing a tweak one click.
Cons: always shows a modal, even for users who never adjust. The one-click "Cook and forget" flow becomes two clicks.

Option 2 — Inline "adjust" toggle on the recipe page

Add a small pencil icon next to each ingredient that flips it into edit mode for the next cook. Adjustments are visible on the recipe page itself (not in a modal) and persist until you click Cook. After cook, they reset. A small "adjusted" indicator shows which ingredients have been tweaked.

Cons: shares visual space with the existing edit-the-recipe pencil icon, which is a real risk of conflation ("did I just edit the recipe permanently or just for this cook?"). Recipes with ingredient groups still need a separate picker UI, so this only partially replaces CookPicker.

Option 3 — Post-cook "what did you actually use?" review

Cook with the recipe amounts as today. Immediately after the cook succeeds, show a "review what you actually used" panel where the user can adjust the deducted amounts. Stock log entries are amended in place.

Pros: closer to "logging what happened" rather than "planning what will happen." Doesn't add friction to the up-front Cook click.
Cons: requires reverse stock-log edits, which gets messy if any consumed entries have already been depleted to zero (was the entry deleted? do we recreate it?). Users skip post-action prompts at a higher rate than pre-action confirmations.

Recommendation

Ship Option 1. Pre-cook adjustment modal. It matches the workflow (you make the decision before you cook, not after), it reuses the existing CookPicker's modal shell, and the per-row reset link gives a one-click undo for any tweak. The "always shows a modal" cost is real but small — one extra click is the price of honest ledger-keeping.

Reasoning against the alternatives:

  • Option 2 blurs "edit recipe" vs. "adjust for this cook." That distinction is load-bearing — users who think they're tweaking once shouldn't accidentally rewrite the recipe.
  • Option 3 has hard edge cases (consumed entries that hit zero, or that were deleted post-cook). And users skip post-action confirmations.

Behaviour details

What's editable

  • Quantity field per ingredient, pre-filled with the recipe-scaled amount
  • Unit stays whatever the recipe specified (don't let the user change units mid-modal; that's a recipe edit)
  • The "↺ original" reset link per row, shown only when the field has been touched

What's not editable in this modal

  • Ingredient unit (recipe edit territory)
  • Whether the ingredient is "optional" or "skipStockDeduction" (recipe property)
  • Adding new ingredients (recipe edit)

Ingredient groups

When the recipe has groups, the modal merges the group-selection UI from the existing CookPicker with the per-ingredient quantity adjustment. Selected group alternative shows its quantity field; unselected ones are greyed.

Optional ingredients

Show with a "Use this time?" checkbox in addition to the quantity field. Unchecked means the ingredient is skipped entirely for this cook, same as if it weren't in the recipe.

skipStockDeduction ingredients

Show as read-only ("Don't track" badge), not editable, since by definition we're not deducting them.

Stock availability indicator

Same green check / amber out-of-stock indicator that already shows on the recipe page, recomputed based on the adjusted amount, not the recipe amount. So if I bump rice from 100 g to 120 g and that pushes me past my available stock, the indicator flips to amber/red in real time.

API shape

The cook mutation takes an optional ingredientOverrides map keyed by ingredient id. The server uses the override amount instead of the recipe's stored amount when computing deductions. Unspecified ingredients use their stored amounts unchanged. Skipped ingredients (optional, unchecked) pass amount: 0 or are omitted — TBD — see decisions. 

POST /api/recipes/cook
{
  recipeId: string,
  servings?: number,
  groupSelections?: { [groupName]: ingredientId },
  ingredientOverrides?: {
    [ingredientId]: number  // absolute amount in the ingredient's unit
  }
}

Same shape extends to the meal-plan cook endpoint.

Tests

  • Unit test: cook API consumes override amount when supplied, recipe amount when not.
  • Unit test: optional ingredients with 0 override skip consumption entirely.
  • Component test: modal pre-fills with scaled recipe amounts; user edits update local state; submit passes overrides to mutation.
  • Component test: reset link restores the original recipe amount.
  • Component test: group selection + quantity adjustment work together.
  • Integration: real-world flow — cook a recipe with adjusted rice, stock log records the adjusted amount, recipe stored amount unchanged.

Decisions

  1. UX: Option 1 (pre-cook adjustment modal).
  2. "Cook as-is": include it as a secondary action inside the modal — clicking it bypasses any tweaks and cooks with the recipe's stored amounts. Users who never adjust can hit "Cook as-is" with one extra click; the modal still serves as a "what's about to happen" confirmation.
  3. Optional-ingredient skip: send a separate skippedIngredients: string[] array. Cleaner semantics than sentinel amount: 0 values; lets the override + skip concerns stay independent.
  4. "Save these as the new recipe amounts": deferred. Not assuming this is needed for v1.
  5. Unit change: allowed. The quantity field is paired with a unit dropdown showing units that have a conversion path to the ingredient's product default. Changing the unit auto-converts the displayed value; the server receives { amount, unitId } per overridden ingredient.
Additional scope: the meal-plan cook button gets the same modal — Nick uses that path frequently. Both surfaces share the same modal component and the same override-passing pattern; both cook endpoints (/api/recipes/cook and /api/meal-plan-entries/cook) accept the new ingredientOverrides + skippedIngredients fields.

Out of scope

  • "Cook history" feature where you can see what amounts you actually used last 5 times. Useful but separate.
  • Auto-suggesting overrides based on past usage ("you usually use 120 g of rice, not 100 g; tweak the recipe?"). Follow-up.
  • Cross-ingredient validation ("if you cut rice, should we cut everything proportionally?"). Different feature.
  • Mid-cook adjustments — the modal is pre-cook only. If you mess up while cooking, you fix it via the stock-log activity view as today.
