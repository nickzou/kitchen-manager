Operating principles

These constrain every technical choice below. When in doubt, fall back to these rather than industry defaults.

  • Sustainability over speed. No shipping-pressure rhythm. Slower, more deliberate beats fast and brittle.
  • Leverage > capital. Trade technical effort and stubbornness for money. AI fills knowledge gaps that would otherwise require hiring.
  • Small-team thesis. SaaS suffers from organizational cruft because people resort to technical processes to solve organizational problems. Keeping the team small + AI-augmented sidesteps that. Don't choose tooling that only makes sense once the team is 20+.
  • Self-hosted by default. "I don't like Vercel selling me AWS compute back to me." OSS on owned infra unless there's no realistic equivalent (Stripe is the canonical exception). Justify managed-service picks explicitly.
  • No physical office until a partnership specifically demands it.

Where we are today

Kitchen Manager is a single-user, production-deployed personal app at kitchen-manager.waffleruntime.com. The architecture is already multi-tenant (every table scopes by userId, Better Auth issues sessions), so the path to "other people can sign up and use this" is mostly about removing single-user assumptions, hardening the public-facing surface, and adding the commercial plumbing (payments, comms, legal).

What works in our favour:

  • Multi-tenant data model from day one — no schema rewrite.
  • Better Auth + email/password already in place.
  • Full public API mirrors the GUI, with bearer-token API keys per user. That's a meaningful differentiator most kitchen apps don't have.
  • Real-world usage data from your own account — ~30 Baby Spinach purchases, recurring meal plans, real recipes — means edge cases have been exercised.

What doesn't work yet:

  • Zero onboarding. A brand-new user lands on an empty stock view with no context.
  • No payment, no plan, no entitlement system.
  • No account deletion (this is on your ClickUp board as 86e0vjvmw — it's also a legal requirement in Canada under PIPEDA).
  • No transactional email beyond Better Auth's defaults. No support email routing.
  • No marketing site — just the app behind a login.
  • Some performance assumptions break at scale (most list pages fetch everything).
  • The domain name (kitchen-manager.waffleruntime.com) and the product name need work before public launch.

Phases

Each phase has an exit gate. Don't move to the next until the current is done, even if you have time. The instinct will be to build new features — resist it.

Phase 0 · Personal-tool harden · ~2–4 weeks

0. Stop the single-user assumptions from leaking out

Before any other human sees a signup form, fix anything that assumes "the user" means "me."

  • Account deletion flow — user-initiated, cascades through every userId-scoped table. Already on the board (86e0vjvmw); ship it first.
  • Email change + verification.
  • Password reset email through a real provider (Postmark, Resend, AWS SES — Resend is simplest).
  • Tenant-isolation audit — grep every db.select, db.update, db.delete for "is this filtered by userId?" Anything that isn't is a leak.
  • Tenant-isolation tests — one user can't read/write another user's data on every API endpoint. Add a "cross-user" test helper and gate CI on it.
  • Rate limiting on auth and key endpoints. Better Auth has primitives; otherwise upstash/redis is the cheap path.
  • API key scoping — currently a key is full account access. Decide whether you ship that as-is for v1 or add read-only / per-resource scopes.
  • Observability baseline: Sentry (or Highlight) on the front end, structured logs server-side, a single uptime check. You will not catch regressions without these.
  • Backups: confirm what your Postgres host does, when, where backups go, and try a restore once.

Phase 1 · Closed beta · ~4–6 weeks

1. Make it survivable for 10–25 friendly strangers

The goal of this phase is to put it in front of real users (not friends — see GTM doc) and find the bugs and onboarding gaps that you can't see because you built it.

  • Branded name + domain. "Kitchen Manager" is a description, not a brand. Without a name you can't make a logo, can't buy a domain, can't write copy. Pick something and move on. (Naming is a Phase-1 task, not a "later" task — it blocks the marketing site.)
  • Marketing site = unauthenticated routes of the PWA. Decided: don't run a separate WordPress / static site. The marketing surface is built inside the app as unauthed routes (a (marketing) group alongside (app)). Reasons: a single codebase to maintain, SSR via TanStack Start handles SEO, and most importantly interactive product previews are possible inline — the demo problem most B2B SaaS hits (where the WordPress site can't show the real product) goes away. Domain shape: apex resolves to the marketing routes, app.<domain> redirects to the authenticated experience.
  • Sign-up + onboarding flow: account creation, optional first-run wizard (give me 1 stock entry + 1 recipe so the dashboard isn't empty), tooltips or empty-state explanations.
  • Seed data option: "Add starter pantry" button that populates Salt / Pepper / Flour / etc. so new users don't face a blank product list.
  • Loading states across the app (on your board as 86e0zjy0r).
  • Help / docs: short user-facing docs (not the API.md) covering core concepts — stock, products, recipes, meal plans. A single page is enough. Static, in the marketing site or under /help.
  • Support email routed to your inbox. support@<domain>. Reply within 24 hrs in beta.
  • In-app bug report — a "report an issue" link that grabs the current URL + screenshot. Sentry can do this; otherwise a simple form.
  • Legal pages: Terms, Privacy, Cookie. Lawyer-reviewed eventually; for beta a Termly / iubenda template plus disclosure that the service is in beta and may break is fine.
  • Feature flagging for risky changes. Even a hand-rolled boolean column on user is enough — you don't need LaunchDarkly.
Hard truth: the first 5 beta users will surface more blockers than the next 50. Don't open the gates too wide.

Phase 2 · Open beta (free) · ~3–6 weeks

2. Public signup, still free, no payments yet

Open the doors. Goal is to validate the value proposition with people who don't know you. Not to monetize yet.

  • OAuth sign-in at minimum with Google. Drops signup friction by 50%+ for the cooking/meal-prep audience.
  • Pagination + lazy loading for product / recipe / stock-log lists. Some users will have 500+ products. Add a server-side limit before they hit it.
  • Cost monitoring on your hosting bill. Pin the threshold where you'd need to pump the brakes.
  • Newsletter signup on the marketing site so visitors who aren't ready to sign up still convert into an owned channel.
  • Analytics: privacy-friendly (Plausible, Fathom, or self-hosted Umami). You'll want to see signup → first-stock-entry → cook-a-recipe → return-7-days funnels.
  • Mobile pass: ensure every flow works on a phone. You already pay attention to breakpoints; this is a focused pass for parity.
  • PWA install prompt: cheap, lets users add to home screen and feel like a native app. Don't build a native app yet.

Phase 3 · Paid launch · ~4–8 weeks

3. Pricing, payments, and the moment of truth

  • Stripe integration. Use Checkout sessions, not Elements — faster to build, no PCI surface, gives Apple/Google Pay for free.
  • Entitlement model: a single plan column on user ("free", "pro", optionally "lifetime"). Don't overengineer this.
  • Grandfathered beta users: decide. Lifetime free is the easy and friendly call. It also gives you a tier of evangelists.
  • HST/GST handling: as a Canadian sole prop, once you hit ~CAD $30k/yr in revenue you have to register and collect. Stripe Tax handles the mechanics; you handle the registration. Set a calendar reminder at projected $25k.
  • Refund policy. Even a 14-day refund window without questions is huge for trust on first-time buyers.
  • Invoice generation for users who'll expense it. Stripe sends these.
  • Webhook backbone for Stripe events → entitlement table. Make this idempotent.
  • Pricing page on the marketing site.

Phase 4 · Steady state · ongoing

4. Run a small business

  • Weekly time block for support email triage. Hard cap so it doesn't eat the build week.
  • Public changelog / "what's new" page. Drives both retention and "the maker is real and active" signal.
  • Feature flag system that lets paying users opt into experiments (creates a feedback loop without forcing rollouts).
  • Annual security review checklist (run yourself or once you can afford it pay a consultant for a few hours).
  • Quarterly review: are the metrics that matter (see GTM) trending where you want?

Cross-cutting concerns

Legal & compliance (Canada specific)

  • PIPEDA: Canadian federal privacy law. Disclose what data you collect, why, and let users delete it. You're already mostly compliant; the missing piece is documented (Terms + Privacy) and operationalised (account deletion).
  • Quebec Law 25 kicked in 2023. If you have Quebec users you need stricter consent / breach notification language. Worth knowing.
  • HST/GST: the $30k threshold above.
  • EU users: if any sign up, you're in GDPR land. The same Terms/Privacy template covers most of it; the data-subject-access-request workflow is the operational lift.
  • Stripe + Canadian bank: straightforward, but takes a week of paperwork. Start the application early.

Security baseline

  • Argon2 (Better Auth default) for passwords. No legacy to worry about.
  • HTTPS-only, HSTS, CSP headers. You probably have most of these via your host; verify.
  • Don't log secrets. Add a redaction check to your structured logger.
  • Rotation policy for the production API key in CLAUDE.md (rotate it now; it's been in the file for weeks). For new tokens, treat them as secrets in 1Password or similar — not in CLAUDE.md.
  • Dependabot or Renovate for dependency updates. You won't have time to chase these manually.

Blindspots & risks

The scope creep instinct is the #1 thing that kills indie launches. You've shown an excellent track record of shipping small, reviewable PRs. Carry that discipline into product strategy: every "wouldn't it be cool if…" idea between now and Phase 3 launch goes into a backlog, not into the codebase. Default answer for non-essential features: "after launch."
  • Backend gap risk: you flagged this. The biggest server-side cost is going to be tenant isolation bugs — the kind that don't show up in tests because the test data is all one user. The audit + cross-user test pattern in Phase 0 is the cheapest insurance.
  • Single point of failure: you. If you're hit by a bus, the lights go out for users. By Phase 3 you need (a) someone in your life who can pause the Stripe subscriptions and email users in an emergency, (b) written-down runbooks for the 5 things that can break (auth down, payments down, DB down, host down, you're sick), (c) ideally a dead-man-switch arrangement with another solo dev. Don't put this off.
  • Cost of running the database for free-tier users at scale. Set a cap. Have a plan for what happens if 5,000 people sign up in a week (Hacker News effect).
  • Self-hosting interest: the "hypernerd" segment of your ICP will ask "can I run this myself?" Decide now. Options:
    1. No — hosted only. Simplest commercially, alienates the segment that's most likely to evangelise.
    2. Source-available with a license that allows personal use, not commercial resale. Good middle ground for indie products (PostHog, Plausible, Cal.com pattern).
    3. Fully OSS with a paid hosted version (Standard Notes, Tailscale-ish). Most aligned with your "valuable to users" stance, hardest to monetize.
    My pick for you given the no-VC, simple-pricing stance: option 2. You keep the IP, indies get to self-host, you get free word-of-mouth from the segment that talks the most.
  • Mobile: the meal-prepper / shopping audience uses this in the kitchen and the grocery store. PWA is good enough for v1 but the rough edges of "save to home screen" on iOS will bite. Budget a week per phase for mobile polish.
  • Data import: people who already track stuff in spreadsheets or Notion will ask "how do I get my data in?" You don't need a fancy importer; a CSV upload to a few tables would carry you far. Phase 2 nice-to-have.
  • Receipt scanning (on your board as 86e0vp8jy / 86e0vp8j5) is the single feature that turns this from "nerd tool" to "anyone can use it." With the GTM doc's lead-ICP set to the value-conscious cook, this moves closer to the critical path: a value-conscious user who has to type in 30 grocery items by hand will bounce. Cost-effective approach: hosted OCR (Google Vision / AWS Textract) behind a feature flag — one of the few places where managed-service makes sense because the alternative (self-host a Tesseract pipeline) is genuinely worse output for more work. Aim to ship a v1 by end of Phase 2.
  • Onboarding overhead changes radically when the lead ICP changes. The current empty-state experience assumes someone willing to configure quantity units and add their first product. The value-conscious ICP will bounce in 90 seconds if step one isn't either "scan a receipt" or "pick a starter pantry." Phase 1 onboarding work should be sized accordingly — budget more than feels obvious.
  • The product you've built is partly over-engineered for the new lead ICP. Per-product unit conversions, the public API, recursive producible-product nutrition — these are hypernerd-shaped features. They're sunk cost and they don't hurt; but they're also not what wins value-conscious users. Be conscious of not over-investing in that direction at the expense of receipt scanning, expiry tracking polish, and waste-reduction surfacing.

What's not on this roadmap (and why)

  • Native mobile app: PWA first. Native after you have 100+ paying users and a clear ask.
  • Real-time collaboration (multi-user household accounts): big feature, niche-within-the-niche. Phase 5+, only if there's pull.
  • Recipe AI / suggestions / "what can I make with what I have": tempting; cost-and-novelty risk. Defer until you have signal.
  • Integration with grocery delivery (Instacart etc.): huge unlock for the value-conscious segment, but partnership-gated and slow. Park it.
  • Public API marketplace: cool but speculative. Your API is already a differentiator; just publicise it. No need for a marketplace until there's demand.
